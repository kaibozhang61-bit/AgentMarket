# TODO — Agent Marketplace

Date: 2026-03-01

## Future

### Chat history pagination
- Current: entire `history` array stored as one DDB attribute on the session item
- DDB 400KB item limit → ~40-80 exchanges before hitting the wall
- Options:
  - Store messages as separate DDB items (`SK = MSG#{timestamp}`) instead of one array
  - Query with limit for pagination
  - Lazy load on frontend: last 20 messages initially, load more on scroll up
- For LLM context: only send last N messages + summary of earlier ones (full history stays in DDB for display)

### Version history UI
- "Version History" button exists on agent detail page (currently disabled, "Coming soon")
- Backend already archives old versions as `VERSION#{timestamp}` on publish
- Need: UI to list archived versions, view/compare, rollback

### Agent execution run routes
- `RunService` backend fully implemented ✅
- API routes at `/agents/{agentId}/run`, `/agents/{agentId}/runs`, `/agents/{agentId}/runs/{runId}`, `/agents/{agentId}/runs/{runId}/resume` ✅
- Resume endpoint exists but no step type currently produces `waiting_user_input` — will be used when interactive steps are added
- Run history panel on canvas bottom-left with run list + run detail + blackboard state ✅

### Schema validation on create/run
- Validation endpoint exists (`POST /agents/{agentId}/validate`) ✅

- Auto-called before run in `trigger_run` — blocks execution if validation fails ✅
- Note: not called on create — drafts may be incomplete, validation on save would be too strict

- Resume API route exists at `POST /agents/{agentId}/runs/{runId}/resume` (forward compat)
- `RunService.resume_run` and `continue_run` backend logic exists
- Need: a new step type (e.g. `type=input`) that pauses the run with a question
- Need: frontend run detail page with input form when status is `waiting_user_input`
- Use cases: human-in-the-loop approval, collecting additional info mid-execution

### Persistent step-level test history
- Current: single-step test results are ephemeral (returned in API response, not stored)
- Current: full agent runs store stepResults as one embedded list on the AGENT_RUN item
- Problems: can't query by step, 400KB item limit risk, test runs clutter run history
- Future: store step test results as separate DDB items: `SK = STEP_TEST#{timestamp}#{stepId}`
- Enables: "show me all test runs for step-2", step-level debugging history
- Also consider: splitting large AGENT_RUN stepResults into separate items per step

### Similar agents suggestion
- When viewing an `agent` step in the right panel, "Similar Agents" section is a placeholder
- Need: query marketplace for agents with matching inputSchema/outputSchema or similar description

### Deleted agent reference handling
- When a marketplace agent is deleted, composite agents that reference it via `type=agent` steps break silently
- Current: validation catches it before run (returns 400), but the user has no warning until they try to execute
- Need:
  - On agent delete: scan for composite agents that reference it (reverse lookup) and notify authors
  - Or: soft-delete instead of hard-delete — mark as `deprecated`, keep the record, refuse new references but allow existing runs
  - UI: show a warning badge on steps that reference a missing/deprecated agent
  - Consider: prevent deletion if other published agents depend on it (like a foreign key constraint)

### Fuzzy search — Elasticsearch
- Current search: DDB Scan with `contains()` filter on name + description
- Slow and imprecise at scale — no relevance ranking, no typo tolerance
- Replace with Elasticsearch / OpenSearch for full-text fuzzy search
- Index agent name, description, inputSchema field names, tags
- Support relevance scoring, typo correction, synonym matching
